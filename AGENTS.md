# CodeLens 项目开发规范

## 架构规范（强制）

- 开始设计、编码、审查或重构前，必须完整阅读并遵循根目录 [`ARCHITECTURE.md`](./ARCHITECTURE.md)。
- `ARCHITECTURE.md` 是本项目技术栈、前后端边界、DDD 分层、依赖方向、业务边界、命名和注释规范的唯一权威来源；其他文档或既有实现与其冲突时，以其为准。
- 任何实现不得通过跨层导入、跨领域访问基础设施、全局可变容器或供应商类型泄漏绕过架构约束。
- 修改稳定契约、依赖方向、数据所有权、安全边界或部署拓扑时，必须同步更新 `ARCHITECTURE.md`，并按要求新增或更新 ADR。
- 若任务要求与 `ARCHITECTURE.md` 冲突，必须先明确指出冲突并获得架构调整确认，不得静默偏离。

## 编程规范

- Python 使用 3.12，新增代码必须提供完整类型标注，并通过 Ruff 和 mypy 严格检查。
- I/O 密集操作使用异步接口；阻塞调用必须隔离，禁止在事件循环中直接执行长时间同步任务。
- 调用 Git 或外部进程时必须使用参数数组，禁止 `shell=True`，并设置超时、输出上限和明确的允许退出码。
- API 和模型输出在边界处使用 Pydantic 校验；领域层优先使用标准库 dataclass、枚举和值对象。
- TypeScript 启用严格类型检查，禁止使用无理由的 `any`、非空断言和未校验的类型转换。
- 保持文件和函数职责单一。优先使用清晰命名和小型模块，仅在复杂逻辑无法自解释时添加简短注释。
- 配置通过环境变量或配置对象注入，不得在源码中硬编码密钥、模型凭证、仓库路径或环境专用地址。
- 日志和事件使用结构化字段，不得记录 API Key、访问凭证、完整 Prompt、完整模型原始输出或不必要的源代码正文。
- 修改现有代码时保持变更范围聚焦，不顺带重构无关模块，不覆盖或回退用户已有改动。
- 新增依赖前确认现有工具链无法合理解决问题，并锁定依赖版本、补充适配层和契约测试。

## 运行方式

- 后端依赖与命令统一通过 `uv` 管理：

```bash
uv sync --project backend
uv run --project backend codelens-review start
```

- 前端依赖与命令统一通过 `pnpm` 管理：

```bash
pnpm --dir frontend install
pnpm --dir frontend dev
```

- 本机开发默认绑定 `127.0.0.1`。绑定 `0.0.0.0` 时必须配置允许访问的仓库根目录，并将网络视为受信任内网边界。
- 数据库结构使用 Alembic 迁移管理；不得通过手工修改数据库或删除数据来规避迁移问题。
- 后端、Worker 和前端应支持独立启动。开发环境可以组合运行，但不得依赖隐式启动顺序或进程内共享状态。

## 测试与质量门禁

- 行为变更遵循测试驱动开发：先编写能复现需求或缺陷的失败测试，再实现最小修改，最后执行回归验证。
- 测试分为单元、集成、契约、端到端和评测。领域逻辑使用单元测试，Git/SQLite/文件系统使用真实临时夹具，外部模型调用默认使用可注入的假实现。
- 先运行与修改范围直接相关的测试，再运行完整门禁。后端标准命令：

```bash
uv run --project backend pytest backend/tests -v
uv run --project backend ruff check backend
uv run --project backend mypy backend/src
```

- 前端标准命令：

```bash
pnpm --dir frontend test
pnpm --dir frontend build
pnpm --dir frontend exec playwright test
```

- 涉及 Git 范围、`.gitignore`、快照、符号链接或补丁应用的测试必须使用临时真实 Git 仓库，不能只模拟命令输出。
- 涉及并发、任务租约、取消或恢复的测试必须验证重复执行、超时、进程重启和部分失败场景。
- 涉及 UI 的变更必须覆盖桌面和移动视口，检查加载、空数据、失败、部分成功和长文本状态，避免溢出和遮挡。
- 真实 OpenAI、远程 MCP 或网络测试必须显式启用，不得成为默认测试套件的前置条件。
- 未实际运行对应验证命令时，不得声称测试、构建或功能已经通过。
