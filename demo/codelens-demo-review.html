Idempotency guard is checked after the write

src/payments/service.py · lines 84–91

Impact

Two requests carrying the same idempotency key can both persist a charge before either request records the key. Customers may be charged twice when a client retries across workers.

Evidence

82async with self._database.transaction():
83+ charge = await self._gateway.capture(command)
84+ existing = await self._keys.find(command.idempotency_key)
85+ if existing is not None:
86+ return existing.result

hunk_f69c6afc · SHA-256 verified against immutable snapshot

Why this happens

The external capture is performed before checking whether the key already has a completed result. The database transaction cannot roll back the external payment side effect, so the later guard does not provide idempotency.

Recommendation

Reserve the idempotency key atomically before calling the gateway. Persist an in-progress state, return the stored result for completed keys, and make competing requests wait or fail deterministically.
